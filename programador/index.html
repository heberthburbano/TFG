Capa de Datos: Mapeo Resiliente

Mapeo resiliente asíncrono desde Firestore (documentToSpecies), evitando fallos en UI (NullPointerException) si un campo falta localmente usando The Elvis Operator (?:). 

private fun documentToSpecies(doc: DocumentSnapshot): Species {
    val data = doc.data ?: return Species()
    return Species(
        id = doc.id,
        scientificName = data["scientificName"] 
            as? String ?: "",
        rarezaMensual = data["rareza_mensual"] 
            as? Map<String, String> ?: emptyMap(),
        migratory = data["migratory"] as? Boolean ?: false
    )
}

Estructura de Firebase NoSQL

El ecosistema Backendless de Firebase almacena avistamientos, información biológica estática de las especies y ficheros multimedia en Storage.

Colecciones Principales

  • users/{userId}: Datos OAuth, estadísticas, y subcolección discoveries/.
  • posts/{postId}: Feed social (latitude, longitude, visibility, etc). Subcolección interna likes/.
  • species/{speciesId}: ViraPedia. Nombres en 5 idiomas (ES, EN, FR, DE, ZH), diet, conservationStatus y propiedades taxonómicas completas.
Reglas de Seguridad: La colección species es de solo lectura pública. El scraper no sobreescribe ciegamente para mantener la integridad relacional de la DB.

Gestión de Rate Limits (Crawler Backend)

Los scripts en Python deben controlar transaccionalmente el volumen de peticiones por segundo. Implementación crítica de extracciones Regex para calcular tiempos de espera dinámicos al enfrentar códigos 429 RESOURCE_EXHAUSTED en la API.

except Exception as e:
    error_msg = str(e)
    # 429 Error: Servidor saturado
    if "429" in error_msg or "RESOURCE_EXHAUSTED" in error_msg:
        delay = 30
        match = re.search(r'retry in (\d+\.?\d*)s', error_msg.lower())
        if match:
            delay = float(match.group(1)) + 1
        
        print(f" [!] Rate limit. Wait {delay:.1f}s...")
        time.sleep(delay)
    else:
        return None

Reconocimiento Asíncrono: BirdNET (Edge AI)

Integración de TensorFlow Lite para identificar cantos 100% offline dentro de BirdNetAudioRecognizer.kt.

Flujo de Inferencia

  1. Grabación nativa de 3 segundos en PCM_16BIT a 48kHz (micrófono). Aislando el trabajo pesado en Dispatchers.IO.
  2. Acondicionamiento: Normalización pura / 32768.0f cargando el bloque en un FloatArray.
  3. Inferencia en birdnet.tflite (~25MB), resolviendo el argmax de clases resultantes.
suspend fun startRecordingAndAnalyze(ctx: Context): Result<String> = 
    withContext(Dispatchers.IO) {
    
    // 1. Raw Microphone recording
    val pcmData = ShortArray(requiredSamples)
    
    // 2. Normalization for TFLite Engine
    val inputFloatArray = FloatArray(requiredSamples)
    for (i in pcmData.indices) {
        inputFloatArray[i] = pcmData[i] / 32768.0f
    }
    
    // 3. Tensor Execution (Edge AI)
    interpreter!!.run(arrayOf(inputFloatArray), outputArray)
    val probabilities = outputArray[0]
}

Modelos Generativos: Gemini y Vertex AI

Integración de LLMs avanzados a nivel Backend (Python) y a nivel cliente Frontend (Kotlin). La plataforma se apoya en el ecosistema Vertex AI por motivos arquitectónicos críticos.

¿Por qué Firebase Vertex AI en lugar de Google AI Studio?

Aunque ambos utilizan el modelo "Gemini", el SDK utilizado dicta el nivel de producción de la app. La elección de Vertex AI se basa en tres pilares irrenunciables:

  1. Seguridad Extrema (Zero API Keys): Usar AI Studio directo requiere incrustar un API_KEY en el código cliente de Android, susceptible a piratería mediante ingeniería inversa (descompilación). El SDK de Firebase Vertex AI emplea App Check, validando criptográficamente que la petición proviene exclusivamente de una instalación legítima del móvil, sin exponer ningún secreto.
  2. Privacidad Enterprise: La capa gratuita de AI Studio se reserva derechos para entrenar modelos públicos con tus prompts. Vertex AI Cloud opera bajo acuerdos empresariales donde ningún dato se utiliza para entrenamiento externo, garantizando 100% privacidad de los avistamientos y reportes.
  3. Sinergia con el Ecosistema: Al usar Firestore, Vertex AI nos permite gobernar cuotas, bloqueos de seguridad y métricas bajo el mismo panel unificado, acoplado al `userId` del módulo Auth nativamente.

Enriquecimiento Python (enrich_bird_details.py)

Script encargado de poblar sistemáticamente la base de datos Firestore empleando Gemini 2.5 Flash con un nivel estricto de determinismo (temperature=0.1 y response_mime_type="application/json"). Sus cometidos principales son la extracción de parámetros biológicos (peso, hábitat, dimorfismo) y la traducción simultánea a 4 idiomas adicionales (ES, FR, DE, ZH) para todas las especies.

Rotación de API Keys y Resiliencia

Para sortear la saturación de Rate Limits (HTTP 429), el código implementa un mecanismo de rotación ("Round-Robin") con múltiples API_KEYS. Al agotar la cuota de una cuenta de Google GenAI, pivota automáticamente a la siguiente clave de respaldo, logrando procesamientos masivos sin interrupciones.

Criptografía y Mensajería E2EE

La mensajería uno-a-uno en ViraPedia está protegida mediante encriptación de extremo a extremo (E2EE), garantizando que los servidores centrales (Firestore) actúen solo como intermediarios ciegos.

El Núcleo Criptográfico: CryptoManager.kt

Las claves se generan localmente en el dispositivo utilizando el algoritmo RSA/ECB/PKCS1Padding. La clave privada se custodia fuertemente encriptada mediante la librería de Google androidx.security.crypto.EncryptedSharedPreferences respaldada por una Master Key AES256_GCM almacenada en el hardware Android Keystore.

fun encryptMessage(message: String, base64PublicKey: String): String {
    val publicKey = getPublicKeyFromString(base64PublicKey)
    val cipher = Cipher.getInstance("RSA/ECB/PKCS1Padding")
    cipher.init(Cipher.ENCRYPT_MODE, publicKey)
    val encryptedBytes = cipher.doFinal(message.toByteArray(Charsets.UTF_8))
    return Base64.encodeToString(encryptedBytes, Base64.NO_WRAP)
}

fun decryptMessage(encryptedBase64Message: String): String {
    // La clave privada nunca viaja por la red (Local-Only)
    val privateKey = getMyPrivateKey() ?: throw IllegalStateException("Private key not found")
    val cipher = Cipher.getInstance("RSA/ECB/PKCS1Padding")
    cipher.init(Cipher.DECRYPT_MODE, privateKey)
    val encryptedBytes = Base64.decode(encryptedBase64Message, Base64.NO_WRAP)
    return String(cipher.doFinal(encryptedBytes), Charsets.UTF_8)
}

Doble Encriptación en Firestore

En DirectMessageRepositoryImpl.kt, cuando el usuario A envía un mensaje al usuario B, el texto se encripta dos veces por separado antes de realizar un Batch Write a la base de datos:

  1. Para el receptor: Se cifra usando la clave pública de B (receiverPublicKey).
  2. Para el emisor (historial): Se cifra usando la clave pública de A (senderPublicKey), guardado en senderCopyContent.

De esta manera, la base de datos Firebase solo almacena blobs de Base64 indescifrables.

Notificaciones On-The-Fly

Al recibir un Push de Firebase Cloud Messaging por parte del servidor, FcmService.kt se ejecuta en un Work Thread (sin abrir la UI de la app).

Intercepta el encryptedContent que trae el JSON de la notificación, invoca transparentemente a CryptoManager.decryptMessage() local, y finalmente eleva una notificación en texto plano directamente al Notification Manager de Android nativo.

Key Escrow (Backup Seguro): Para evitar que el usuario pierda acceso permanente a su historial de chat multianual en caso de un formateo del terminal, CryptoManager.kt provee métodos de wraping AES (`getOrGenerateAesKey`, `wrapPrivateKey`). Las llaves envueltas se ligan al framework de Backups Nativos en la Nube de Android (backup_rules.xml).

Configuración del Entorno

El proyecto requiere un entorno específico debido a las múltiples llaves de APIs biológicas y al uso del Edge AI (TFLite).

Requisitos del Sistema

  • IDE: Android Studio Jellyfish (o superior).
  • JDK: JDK 17 (Integrado en el proyecto).
  • SDK de Android: Mínimo API 24, Target SDK 34+.

Secretos y Variables (`local.properties`)

El manifiesto (AndroidManifest.xml) requiere un token válido para el SDK de mapas que deberá inyectarse en el local.properties:

MAPS_API_KEY=AIzaSyTuClaveSecretaDeGoogleMaps...

Integración Firebase (`google-services.json`)

Debes colocar el archivo google-services.json en la raíz del módulo app/ para que el generador de Hilt/Firebase provea los repositorios de la base de datos (Firestore) y del sistema Auth.

Guía de Compilación

El compilador asume un acoplamiento puro con `gradlew` y módulos inyectados.

1. Limpieza y Reconstrucción

Tras inyectar los Secrets de las APIs y Firebase, ejecuta ./gradlew clean build para forzar a Hilt a re-generar todos los grafos de dependencias Dagger subyacentes.

2. Ejecución (Debug)

Haz click en Run 'app'. Recomendamos encarecidamente testear la inferencia de Puntos de Navegación y Grabación `PCM_16BIT` en un dispositivo físico, ya que el emulador virtual de Android a menudo no simula las frecuencias del micrófono requeridas por BirdNET.

Cuidado con los Permisos: Asegúrate de que el manifiesto solicita activamente los permisos RECORD_AUDIO, ACCESS_FINE_LOCATION e INTERNET al ejecutar por primera vez.
